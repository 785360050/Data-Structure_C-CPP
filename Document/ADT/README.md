# ADT Documents

本目录用于存放项目内各抽象数据类型的正式定义。

这些文档的定位是：

- 作为同类数据结构实现的公共契约来源
- 用于区分 ADT 公共语义与具体实现特性
- 作为共享测试设计和接口评审的依据

## 目录分类

本目录按照项目内数据结构类型分类组织，当前分为：

- `Linear_Structure/`
- `Tree_Structure/`
- `Graph_Structure/`
- `Set_Structure/`

## 使用规则

1. 每个 ADT 文档只定义该 ADT 的公共逻辑语义。
2. 不把具体存储实现细节写进 ADT 公共契约。
3. 共享测试优先依据本目录文档编写，而不是依据某个具体实现。
4. 头文件中只保留简要摘要；完整定义以本目录文档为准。

## 写法约定

1. 正式 ADT 文档只保留 ADT 本体定义，不写过程性说明。
2. 不写实现映射、测试建议、评审评论等非契约内容。
3. 操作部分采用按需描述的接口说明风格，只写该操作真正需要的项目。
4. 不强制为每个操作补齐“无”的字段。
5. 接口名直接写在对应小节标题中，例如：
   - `插入元素 Element_Insert(pos, elem)`
   - `删除元素 Element_Delete(pos)`
6. 一致性部分只约束公共行为，不限制具体实现方式。

## 当前推荐结构

基于当前仓库的写法，正式 ADT 文档推荐使用以下结构：

1. 定义对象
2. 公共状态语义
3. 位置约定或参数约定
4. 公共操作与契约
5. 一致性要求

## 代码组织说明

在当前仓库中，一个 ADT 目录下可能同时存在两类文件：

1. `XXX.hpp`
   - 用于定义当前实际使用的逻辑抽象层
   - 通常包含抽象基类、共享状态、纯虚接口或默认行为
   - 具体实现一般直接继承这一层

2. `ADT.hpp`
   - 用于定义接口约束层
   - 当前通常承载 C++20 `concept` 之类的接口检查
   - 某些目录中也可能暂时保留废弃的旧接口定义

以 `Linear_List` 为例：

- `Linear_List.hpp`：当前实际使用的逻辑抽象层
- `ADT.hpp`：接口约束层，同时混有历史遗留接口

因此，正式 ADT 文档应优先描述“这个 ADT 的公共语义”，而不是直接照抄目录中的某个头文件结构。
