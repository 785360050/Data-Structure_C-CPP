# 可视化 MVP 实验结论

更新时间：2026-04-04

## 一、实验目标

本轮实验的目标不是建立完整可视化框架，而是验证下面三个问题：

1. 当前仓库是否能在尽量少改 `C++` 主体实现的前提下做出可运行的可视化原型
2. 是否能在同一界面中同时表达 `ADT` 语义和具体实现细节
3. 这条路线对后续复杂数据结构是否可持续

## 二、本轮实现范围

本次仅选择最简单、最稳定的一类结构作为最小可行验证对象：

- `Sequential_List_Static<int, 10>`

实现位置：

- [Linear_List.hpp](/mnt/WorkSpace/Github/Data-Structure_C-CPP/QT/Structure/Linear_Structure/Linear_List.hpp)
- [Linear_List.cpp](/mnt/WorkSpace/Github/Data-Structure_C-CPP/QT/Structure/Linear_Structure/Linear_List.cpp)

本轮已经实现：

- `Insert(position, value)` 的逐帧回放
- `Delete(position)` 的逐帧回放
- `Monitor` 页签中的 `Prev / Next / Play / Reset`
- 每一帧同时显示：
  - `ADT` 语义说明
  - `Implementation` 实现说明

## 三、实现方式

本次采用的是低侵入方案，而不是直接改核心顺序表实现。

具体做法是：

1. 真实容器仍然使用现有 `Sequential_List_Static`
2. Qt 侧额外维护 `Frame` / `Slot` 结构
3. 操作前先抓当前状态
4. 根据顺序表语义在外部推导插入或删除过程
5. 生成逐帧 `trace`
6. 调用真实容器执行操作
7. 使用 Qt 播放生成的帧

这说明：

- 第一版可视化不一定要侵入核心实现
- 对规则明确的结构，可以先用“外部推导 + 最终状态校正”完成 MVP

## 四、本轮得到的核心结论

### 1. 方案可行

当前仓库可以在不重写主线 `C++` 结构实现的前提下做出可运行的可视化原型。

### 2. “ADT 层 + 实现层”可以同时表达

顺序表的插入和删除验证了一个关键点：

- `ADT` 层可以描述“逻辑序列发生了什么变化”
- `Implementation` 层可以描述“底层顺序存储做了哪些搬移和写入”

所以“同一抽象数据类型下体现不同实现方案特点”在设计上是成立的。

### 3. 真正昂贵的不是绘图，而是中间表示

这次工作量主要不在“画矩形”，而在：

- 状态快照建模
- 操作拆帧
- 高亮规则
- 文本说明
- 回放控制

也就是说，后续成本主要来自：

- 如何把数据结构状态表达成统一协议
- 如何把一次操作拆成可播放步骤

### 4. 如果按“每个结构单独手搓页面”推进，后续会失控

顺序表之所以容易，是因为它具备：

- 固定布局
- 简单状态
- 明确步骤
- 可外推过程

但复杂结构不会这么简单。

## 五、对后续复杂结构工作量的判断

如果继续按“一个结构一套特化可视化”推进，工作量会很大，而且会越来越大。

大致可以分成几档：

- 中低成本：
  - `Stack`
  - `Queue`
  - `Binary_Heap`
- 中等成本：
  - `Link_List`
  - `Skip_List`
- 高成本：
  - `BST`
  - `AVL`
  - `Red_Black_Tree`
- 很高成本：
  - `B-Tree`
  - `B+Tree`
  - `R-Tree`
- 极高成本：
  - `Graph`

原因不只是绘制更复杂，而是：

- 状态结构更复杂
- 操作步骤更长
- 布局稳定性更难处理
- 高亮与解释规则更复杂

## 六、前端是否能消除这个问题

不能。

本轮实验说明，问题的本质不是 `Qt` 还是前端，而是：

- 是否存在统一的 `Trace / Snapshot / Renderer` 分层
- 是否按少数几种结构原型复用渲染器
- 是否控制展示粒度

前端的优势在于：

- 更适合做通用播放器、面板、对比视图
- 更适合长期组件化和分享部署

但前端并不能自动消除下面这些成本：

- 为不同结构建模状态
- 为不同操作拆分步骤
- 为不同实现表达特有行为

所以：

- 前端可以改善长期复用效率
- 但不能消除“结构建模”本身的工作量

## 七、当前阶段的技术判断

按当前能力结构和仓库状态，主线方案仍应保持为：

- `C++` 主导
- 先抽离可视化中间协议
- 先用 Qt 验证方案
- 前端作为后续可选渲染器
- `VTK / vtk.js` 留给空间结构与 3D 可视化阶段

这比立即切到前端或立即切到 `VTK` 更稳。

## 八、后续建议

本轮实验之后，不建议立刻继续做新的复杂结构页面，而是先做两件事：

1. 抽公共层
   - 把当前 `Linear_List` 中的 `Frame / Slot / TraceBuilder` 思路抽成独立基础层
2. 做第二个对照实现
   - 优先选择 `Link_List`

这样做的价值是：

- 验证“同一 ADT，不同实现”的方案是否真的可复用
- 避免项目退化成“每个结构各写一套 UI”

## 九、结论摘要

本轮实验最重要的结论有三点：

1. 可视化方案在当前仓库中是可落地的
2. `ADT` 语义和实现细节可以同时展示
3. 后续真正需要尽快抽象的不是绘图代码，而是统一的 `Trace / Snapshot / Renderer` 中间层

如果不做这层抽象，后续复杂结构的可视化工作量会迅速膨胀。
