# 测试完善规划

更新时间：2026-04-06

## 一、当前决策

项目后续长期保留 `Boost.Test`，不计划迁移到 `GoogleTest`。

当前判断依据如下：

- 本项目现有测试代码已经使用 `Boost.Test`
- 对当前仓库的主要测试需求而言，`Boost.Test` 的能力是足够的
- `Boost.Test` 与系统包管理结合更自然，维护成本更低
- 本项目当前更需要“继续扩大默认回归覆盖”，而不是“更换测试框架”

因此，后续测试工作的重点是：

> 在保留 `Boost.Test` 的前提下，逐步把测试体系工程化、系统化、可回归化。

## 二、测试建设总目标

后续测试部分不是简单“多写几个测试文件”，而是要逐步建立下面这套能力：

1. 测试可统一构建
2. 测试可统一执行
3. 测试覆盖核心数据结构
4. 测试覆盖正常路径与边界路径
5. 测试覆盖拷贝/移动/析构等对象语义
6. 每次修 bug 都能沉淀为回归测试

## 三、后续总体策略

测试完善按以下顺序推进：

1. 先建立统一测试基线
2. 再补基础结构测试
3. 再补高级结构测试
4. 最后补质量增强层

这里“质量增强层”包括：

- Sanitizer
- Valgrind
- 编译警告清理
- 覆盖率统计

## 七、每个模块应覆盖的测试类型

后续每个数据结构模块，尽量按以下五类测试补齐。

### 1. 接口行为测试

验证基础功能是否正确，例如：

- 插入
- 删除
- 查询
- 更新
- 清空
- 遍历
- 长度变化

### 2. 边界条件测试

这是最重要的一类。至少应覆盖：

- 空容器操作
- 单元素容器操作
- 满容量操作
- 非法位置访问
- 越界插入 / 删除
- 重复插入 / 重复删除
- 扩容 / 缩容边界

### 3. 对象语义测试

本项目手写了较多拷贝控制和移动语义，这类测试必须长期保留。

至少应覆盖：

- 拷贝构造
- 拷贝赋值
- 移动构造
- 移动赋值
- moved-from 对象是否合法
- 是否存在资源泄漏 / 双重释放 / 共享错误

### 4. 结构不变量测试

对数据结构项目来说，这类测试价值很高。

例如：

- 堆：父子节点满足堆序
- AVL：任一节点平衡因子合法
- 红黑树：颜色规则与黑高规则成立
- 并查集：连通关系正确
- 图：顶点、边、遍历结果与存储状态一致

### 5. 回归测试

以后每发现一个 bug，都补一个最小复现测试，再修复实现。

这是长期稳定质量的关键。

## 十一、后续工作原则

等正式开始处理测试部分时，建议遵守以下原则：

1. 先把已有测试纳入统一流程，再新增测试。
2. 每补一个模块，就尽量补齐“行为 + 边界 + 对象语义”三类测试。
3. 每修一个 bug，就补一个回归测试。
4. 优先把基础容器测试打牢，再处理高级结构。
5. 测试命名、目录结构、目标命名尽量统一。

## 十二、测试分层规划

后续测试不建议继续混成一类，而是分为三层：

### 1. ADT 契约测试

这一层只测试所有实现都应满足的公共语义，不绑定具体存储方式。

例如：

- 线性表初始化后为空
- 插入后元素数量变化正确
- 越界访问抛异常
- 删除后元素顺序正确
- 清空后恢复到初始逻辑状态

这一层适合做成模板化共享测试，复用于：

- `Sequential_List_Static`
- `Sequential_List_Dynamic`
- `Link_List_Forward`
- `Link_List_Double`

以及后续其他满足同一 ADT 的实现。

### 2. 实现特性测试

这一层只测试某种实现独有的行为，不应误当成所有实现的统一契约。

例如：

- 静态顺序表满容量后的行为
- 动态顺序表的扩容/缩容策略
- 链表实现的特定边界行为
- 堆、树、图各自的内部结构不变量

这类测试可以长期保留，但不应该写进 ADT 公共测试中。

### 3. 回归测试

每修复一个已确认 bug，就补一个最小复现测试。

这一层的作用是：

- 防止同类问题再次出现
- 把历史缺陷转化为长期质量资产

回归测试有时属于 ADT 层，有时属于具体实现层，需要按问题本身归类。

## 十二、关于“特殊行为”的处理原则

后续测试中，不应默认把当前实现的所有特殊行为都固化为长期契约。

建议区分为三类：

1. 公共接口语义
   - 这类行为应写入测试并长期稳定保证
2. 当前实现细节
   - 这类行为可以记录，但不应轻易写死到公共测试中
3. 已知不健康或待修行为
   - 不应继续作为正确行为固化到测试里

例如：

- “越界抛异常”属于公共语义
- “扩容倍率恰好是多少”更偏实现细节
- “moved-from 对象容量必须为 0” 是否合理，需要单独判断，不应默认作为统一契约

## 十三、当前阶段优先级

## 十五、2026-04-05 测试设计重审结论

本节用于记录当前阶段对测试设计的重新审视结论，作为后续重构测试实现时的直接依据。

### 1. 当前测试体系的定位

当前 `Data_Sturcture/C++` 测试体系已经具备：

- 可统一构建
- 可统一执行
- 有一批基础模块进入默认验证主线

但当前体系仍主要处于“最小可回归入口”阶段，还不能视为已经完成系统化测试设计。

当前测试更接近：

- 基础行为验证
- 冒烟验证
- 部分对象语义验证

而不是：

- 系统化功能测试
- 系统化边界测试
- 缺陷驱动的回归测试体系

### 2. 对当前测试设计的判断

当前测试可以满足一部分功能验证，但不能认为已经充分满足“功能测试 + 回归测试”的设计目标。

具体判断如下：

1. 功能测试：部分满足
   - 已覆盖一批基础结构的正常路径
   - 已覆盖少量异常路径和对象语义
   - 但边界条件覆盖仍不系统
2. 回归测试：尚未系统建立
   - 当前缺少“每个已确认 bug 对应一个最小复现测试”的沉淀方式
   - 当前也缺少独立的回归测试分层和命名约定
3. 测试实现方式：需要重构
   - 当前各测试文件风格不统一
   - 公共辅助逻辑重复较多
   - 一部分断言绑定了具体实现细节，不利于长期维护

### 3. 重构的核心目标

测试重构的重点不是“多写一些 case”，而是先把测试组织方式设计正确。

本轮重构的核心目标是建立一套可以长期扩展的测试结构，使其同时支持：

- ADT 级公共行为验证
- 实现级特性验证
- 缺陷驱动的回归测试
- 后续高级结构的平滑接入

### 4. 设计原则

后续测试重构遵循以下原则：

1. 优先测试公共接口契约，而不是实现细节
2. 先把测试分层，再补测试数量
3. 能抽象到 ADT 层的测试，优先做成共享测试样板
4. 实现特有行为只放在实现层测试中
5. 每个已确认 bug 都沉淀为最小回归测试
6. moved-from 对象只验证“仍合法”，不默认绑定具体内部状态

## 十六、重构后的测试分层设计

后续测试建议明确分为四层。

### 1. Behaviour

这一层负责验证 ADT 的公共行为契约。

它应覆盖：

- 初始化后的逻辑状态
- 插入 / 删除 / 更新 / 查询 / 清空
- 元素顺序变化
- 长度变化
- 公共异常行为

这一层的目标是：

- 承担功能测试的主体
- 为同一 ADT 的多个实现复用同一套测试样板

例如：

- 栈的 LIFO 行为测试可同时复用于顺序栈和链栈
- 队列的 FIFO 行为测试可同时复用于顺序队列和链队列
- 线性表的插入 / 删除 / 访问语义可同时复用于顺序表和链表

### 2. Boundary

这一层负责验证边界路径和异常路径。

它应覆盖：

- 空容器操作
- 单元素状态转换
- 满容量行为
- 越界访问
- 非法插入 / 删除 / 更新
- 扩容 / 缩容边界
- 环形结构的绕回边界

这一层仍然优先从 ADT 语义出发，但允许补充少量实现相关边界。

### 3. Object_Semantics / Invariants

这一层负责验证对象语义与结构不变量。

对象语义包括：

- 拷贝构造
- 拷贝赋值
- 移动构造
- 移动赋值
- 析构安全
- moved-from 对象是否合法

结构不变量包括：

- 堆序是否成立
- AVL 平衡因子是否合法
- 红黑树颜色规则是否合法
- 并查集连通关系是否一致

这一层通常不适合完全抽象为纯 ADT 测试，因为它往往依赖具体实现能力。

### 4. Regression

这一层负责沉淀历史缺陷。

基本原则：

- 每修复一个已确认 bug，就补一个最小复现 case
- 回归测试优先最小化，不与大而全的功能测试混写
- 回归测试可以属于 ADT 层，也可以属于具体实现层

区分规则如下：

1. 如果 bug 破坏的是公共契约
   - 应补到 ADT 层的 behaviour 或 boundary 测试中
2. 如果 bug 只属于某个实现
   - 应保留在该实现自己的 regression 测试中

## 十七、ADT 共享测试的定位

基于 ADT 复用同一套测试，是后续测试设计的主轴之一。

### 1. ADT 共享测试属于什么层

ADT 共享测试本质上主要属于：

- `behaviour`
- `boundary`

它验证的是：

- 某个抽象数据类型应满足的公共接口契约
- 不同实现是否都遵循同一套语义规则

### 2. ADT 共享测试能承担什么

ADT 共享测试可以承担：

- 大部分通用功能测试
- 一部分通用边界测试
- 一部分最终上升为公共契约的回归测试

### 3. ADT 共享测试不能替代什么

ADT 共享测试不能替代：

- 实现特有的回归测试
- 资源管理层面的特殊缺陷复现
- 高级结构依赖内部不变量的测试

例如：

- 顺序队列环形下标绕回错误
- 动态顺序表扩容搬移策略错误
- 某个 move 赋值路径下的双重释放

其中有些最终可以提升为 ADT 边界测试，有些则应保留为实现层回归测试。

### 4. 结论

因此，ADT 共享测试的定位应明确为：

- 功能测试的主体
- 通用边界测试的主体
- 通用回归测试的吸收层

而不是：

- 全部测试的唯一承载层

## 十八、当前确定的目录结构方案

结合当前仓库规模与迁移成本，测试目录不再继续做过深分层，而采用“一个模块一个文件”的收敛方案。

建议的主目录结构如下：

```text
Data_Sturcture/C++/Test/
  Linear_List/
    Sequential_List.cpp
    Link_List.cpp
  Stack/
    Sequential_Stack.cpp
    Link_Stack.cpp
  Queue/
    Sequential_Queue.cpp
    Link_Queue.cpp
  Tree/
    Binary_Heap.cpp
  Unit_Test/
    ...
```

其中：

- 新目录用于承接重构后的测试
- `Unit_Test/` 暂时保留，作为迁移过渡层
- 已经迁移稳定的模块，后续再考虑是否替换旧测试

该结构的核心目标是：

- 控制目录层级
- 控制文件数量
- 保持模块定位清晰
- 降低后补测试时的维护成本

### 1. 文件内部的分层方式

虽然一个模块只使用一个测试文件，但文件内部仍要按 suite 分层，不能重新退化成“大杂烩”。

推荐形式如下：

```text
Sequential_List
  Behaviour
  Boundary
  ObjectSemantics
  Regression
```

也就是说：

- 物理上：一个模块一个文件
- 逻辑上：在同一文件内按测试层次分 suite

### 2. 当前阶段的最低要求

当前阶段至少要求每个已重构模块满足：

- `Behaviour`

其他层按实际情况逐步补充：

- `Boundary`
- `ObjectSemantics`
- `Regression`

这符合当前项目状态：

- 先把 ADT 公共行为契约测稳
- 不强行补齐所有历史回归
- 后续发现 bug 再持续沉淀回归测试

### 3. 为什么采用单文件方案

不采用更细目录和多文件拆分的原因是：

- 当前仓库仍处于后补测试阶段
- 过深目录和过多小文件会增加维护成本
- 现阶段更重要的是先把测试语义组织正确，而不是先把物理结构拆到最细

因此，当前正式记录的方案是：

> 每个模块一个测试文件，文件内部用 suite 区分 `Behaviour`、`Boundary`、`ObjectSemantics`、`Regression`。

## 十九、当前阶段的执行建议

2. 先把现有 case 按 `Behaviour / Boundary / ObjectSemantics / Regression` 重新归类
3. 每个模块先收敛到一个新的测试文件
4. 去掉对实现细节的硬编码断言
5. 再逐步推广到栈、队列、并查集、搜索树、图

## 二十、当前记录的明确结论

本轮设计重审后的明确结论如下：

1. 当前测试尚未完整达到“系统化功能测试 + 系统化回归测试”的目标
2. 当前测试实现需要重构，但框架仍保留 `Boost.Test`
3. 当前阶段测试至少满足 `Behaviour` 层
4. ADT 共享测试应作为功能测试主体
5. 回归测试仍需独立分层，但在文件内部组织即可，不强制拆成独立文件
6. 后续测试扩展应优先围绕 ADT 契约与缺陷沉淀两条主线展开

## 二十一、可视化与 Trace 层的当前计划

当前阶段对可视化、调试辅助与数据结构实现之间的关系，记录如下：

### 1. Trace 层的定位

`Trace` 层不是 `Qt` 专用层，而是位于数据结构实现与表现层之间的中间表示层。

它应同时服务于：

- 开发期调试
- 控制台过程输出
- 单元测试辅助观察
- `Qt` 可视化消费
- 后续可能的前端可视化消费

因此，后续设计目标不是“直接把 `Qt` 逻辑耦合进数据结构实现”，而是先建立统一的 `Trace / Snapshot / Renderer` 分层。

### 2. 当前阶段的侵入原则

当前阶段已经明确以下边界：

- 不在数据结构对象中放 `trace` 成员
- 不修改数据结构构造函数以接入 `trace`
- 不让 `ADT` 层知道 `trace`
- 不允许 `Qt`、`Frame`、`Slot`、播放器状态等展示概念进入数据结构实现层

这意味着，当前主张不是“统一地把 `trace` 注入所有数据结构实现”，而是优先保持核心实现纯净。

在此基础上，仅允许如下级别的接入：

- 简单结构优先使用外部包装与外部推导
- 复杂结构仅在少数关键私有步骤旁边放极少量 hook
- hook 的改动规模应尽量接近“打一行日志”
- 未开启 `trace` 时，数据结构仍按普通方式使用

也就是说，当前阶段的目标更准确地应表述为：

- 展示层零侵入
- 核心实现尽量零侵入
- 复杂结构在必要时采用极轻量 hook

### 3. 当前阶段不采用宏方案

当前约定不使用宏来实现 `trace` 开关与事件上报。

原因是：

- 宏会引入额外的预处理分支
- 不利于类型检查、重构与调试
- 容易使代码出现多套行为路径

后续 `trace` 机制应优先采用现代 C++ 的抽象与对象组合方式实现。

### 4. 当前建议的实现方向

当前建议的实现方向已调整为“两类结构、两种策略”：

1. 简单结构

- 使用外部包装与外部推导生成 `trace`
- 典型对象包括：顺序表、普通栈、普通队列
- 这类结构不修改核心实现
- 当前顺序表最小案例中，`Trace` 层只保留快照、帧与 `Recorder`，步骤文案与推导逻辑全部放在 `Qt` 包装层
- 当前顺序表最小案例中，`Qt` 的帧标题与说明文字已改为复用 `Trace` 层生成逻辑，不再单独维护一套英文文案

2. 复杂结构

- 在实现内部的关键私有步骤旁边放极少量 hook
- 典型对象包括：`AVL`、红黑树、`B/B+Tree`、图等
- hook 仅用于暴露外部无法可靠推导的关键过程

后续若需要为复杂结构接入调试型 `trace`，优先采用：

- 独立的 `Session`
- 独立的 `Recorder`
- 线程局部的当前 hook 上下文

也就是说：

- 数据结构对象本身不保存 `trace` 状态
- 外部通过作用域方式临时开启 `trace`
- 实现层只在极少量关键点查询当前 hook 是否存在
- 未启用时不影响普通使用
- 当前顺序表已经用 `Session + Recorder + Console` 跑通了最小开发期调试链路，调试描述统一使用中文输出
- 当前顺序表的控制台输出与 `Qt` 帧文本已经共享同一套 `Trace` 描述来源
- 当前记录进一步分为两级：
- `Debug` 级别：不依赖 hook，主要记录操作前后快照，定位为日志式观察
- `Trace` 级别：依赖少量关键 hook，主要记录扩容、缩容、搬移、写入等中间步骤，定位为开发期逐步调试
- 控制台消费者应支持按 `Debug`、`Trace` 或全部进行筛选输出
- 输出模板优先使用 `std::format`，元素值本身仍允许通过流输出转为字符串，以兼容未实现 `std::formatter` 的类型

### 5. 当前阶段的最小目标

当前阶段已经得到的明确结论是：

- 简单结构可先用外部包装承担最小可视化与演示需求
- 顺序表已额外验证“少量关键 hook + 控制台消费者”的最小调试闭环
- 这种方案已经可以承担顺序表当前阶段的开发期调试辅助，但还不等于所有结构的通用调试基础设施
- 如果希望 `trace` 在更复杂结构上继续承担开发调试责任，仍需要按结构补少量关键 hook
- 但该 hook 不应通过成员注入、构造函数改造或 `ADT` 扩展来实现

因此，当前阶段的现实目标调整为：

- 顺序表最小案例继续使用外部包装
- 顺序表保留当前控制台消费者、中文调试输出和 `Debug/Trace` 两级筛选
- 后续从复杂结构中挑一个结构继续验证“关键私有步骤 hook”方案

在此基础上，再评估是否继续推广到其他结构。

### 6. 待考虑点：现有 Show 函数的去向

当前对现有 `Show` / `List_Show` 一类函数的判断如下：

- 不建议直接把 `Show` 函数接入可视化主链路或 `Trace` 主链路
- 更合理的方向是后续逐步改为 `Debug` 快照的消费者，而不是继续作为独立输出体系扩展
- 这样可以让控制台输出、调试快照和后续可视化复用同一份状态来源

当前评估的工作量判断如下：

- 对线性结构，如顺序表、链表、栈、队列，这项改动工作量中等偏小，适合后续推进
- 对树、图等复杂结构，这项改动工作量会明显上升，需要先定义结构化快照模型，再考虑是否收敛现有 `Show`

因此，这一项暂时记录为后续待评估事项，不作为当前阶段必须完成的改动。
