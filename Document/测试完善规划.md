# 测试完善规划

更新时间：2026-04-04

## 一、当前决策

项目后续长期保留 `Boost.Test`，不计划迁移到 `GoogleTest`。

当前判断依据如下：

- 本项目现有测试代码已经使用 `Boost.Test`
- 对当前仓库的主要测试需求而言，`Boost.Test` 的能力是足够的
- `Boost.Test` 与系统包管理结合更自然，维护成本更低
- 本项目当前更需要“把测试体系接通”，而不是“更换测试框架”

因此，后续测试工作的重点是：

> 在保留 `Boost.Test` 的前提下，逐步把测试体系工程化、系统化、可回归化。

## 二、测试建设总目标

后续测试部分不是简单“多写几个测试文件”，而是要逐步建立下面这套能力：

1. 测试可统一构建
2. 测试可统一执行
3. 测试覆盖核心数据结构
4. 测试覆盖正常路径与边界路径
5. 测试覆盖拷贝/移动/析构等对象语义
6. 每次修 bug 都能沉淀为回归测试

## 三、后续总体策略

测试完善按以下顺序推进：

1. 先建立统一测试基线
2. 再补基础结构测试
3. 再补高级结构测试
4. 最后补质量增强层

这里“质量增强层”包括：

- Sanitizer
- Valgrind
- 编译警告清理
- 覆盖率统计

## 四、当前测试基线状态

第一阶段“先把测试入口统一”已经完成最小闭环，当前状态如下：

- 已用 `CMake + CTest` 统一注册现有 `Boost.Test` 测试
- 已建立稳定的测试目录和目标命名规则
- 默认配置下 `ctest` 已可稳定执行一批基础模块
- 当前默认主线已注册并通过 `11` 个测试

当前已纳入默认验证的测试包括：

1. `Sort`
2. `Sequential_List`
3. `Link_List`
4. `Sequential_Stack`
5. `Sequential_Queue`
6. `Link_Queue`
7. `Link_Stack`
8. `Stack_Compatibility`
9. `Node`
10. `Test_Element`
11. `Binary_Heap`

这说明仓库已经不再停留在“手工单文件编译测试”的阶段，而是具备了最小可回归测试入口。

## 五、下一阶段重点

在最小测试基线已经建立的前提下，下一阶段重点变为：

- 给基础容器补更多边界条件测试
- 给手写拷贝/移动语义补对象语义测试
- 每修一个 bug，都补最小回归用例
- 把当前仍未纳入默认主线的高级结构逐步拉入验证范围

## 六、测试编写的优先顺序

后续建议按下列顺序补测试：

1. 顺序表
2. 链表
3. 栈
4. 队列
5. 堆
6. 并查集
7. 二叉搜索树 / AVL / 红黑树
8. 图
9. B 树 / B+ 树

原因：

- 基础线性结构最容易先收敛
- 基础容器是很多高级结构的实现基础
- 当前仓库的资源管理和边界问题也主要集中在基础容器

## 七、每个模块应覆盖的测试类型

后续每个数据结构模块，尽量按以下五类测试补齐。

### 1. 接口行为测试

验证基础功能是否正确，例如：

- 插入
- 删除
- 查询
- 更新
- 清空
- 遍历
- 长度变化

### 2. 边界条件测试

这是最重要的一类。至少应覆盖：

- 空容器操作
- 单元素容器操作
- 满容量操作
- 非法位置访问
- 越界插入 / 删除
- 重复插入 / 重复删除
- 扩容 / 缩容边界

### 3. 对象语义测试

本项目手写了较多拷贝控制和移动语义，这类测试必须长期保留。

至少应覆盖：

- 拷贝构造
- 拷贝赋值
- 移动构造
- 移动赋值
- moved-from 对象是否合法
- 是否存在资源泄漏 / 双重释放 / 共享错误

### 4. 结构不变量测试

对数据结构项目来说，这类测试价值很高。

例如：

- 堆：父子节点满足堆序
- AVL：任一节点平衡因子合法
- 红黑树：颜色规则与黑高规则成立
- 并查集：连通关系正确
- 图：顶点、边、遍历结果与存储状态一致

### 5. 回归测试

以后每发现一个 bug，都补一个最小复现测试，再修复实现。

这是长期稳定质量的关键。

## 八、Boost.Test 计划重点使用的能力

后续优先使用以下能力：

- `BOOST_AUTO_TEST_CASE`
- `BOOST_FIXTURE_TEST_CASE`
- `BOOST_DATA_TEST_CASE`
- `BOOST_CHECK`
- `BOOST_REQUIRE`
- `BOOST_CHECK_THROW`
- `BOOST_REQUIRE_THROW`

后续如果需要按模块筛选测试，还可以利用 `Boost.Test` 的 label / suite 组织能力。

## 九、后续配套质量措施

单元测试之外，后续建议补上以下能力：

### 1. Sanitizer

至少考虑：

- AddressSanitizer
- UndefinedBehaviorSanitizer

原因：

- 本项目大量手写内存管理
- 很多资源管理问题靠普通单测不一定能直接暴露

### 2. Valgrind

Linux 环境下继续保留，用于检查：

- 泄漏
- 重复释放
- 非法访问
- 析构路径问题

### 3. 编译警告治理

后续测试目标建议逐步稳定开启：

- `-Wall`
- `-Wextra`

之后再视情况提高标准。

### 4. 覆盖率

不是第一优先级，但后续可以加入，用来判断：

- 哪些核心模块还没有被真正测到
- 哪些边界路径完全没有经过测试

## 十、建议的下一批样板模块

后续继续扩展测试时，建议优先选以下模块作为下一批样板：

1. `Sequential_List`
2. `Stack` 或 `Link_List`
3. `Binary_Heap`

原因：

- 这几个模块已经有一定测试基础
- 它们能代表当前仓库中最典型的问题类型
- 一旦把这几类测试模式跑通，后面迁移到树和图时会顺很多

## 十一、后续工作原则

等正式开始处理测试部分时，建议遵守以下原则：

1. 先把已有测试纳入统一流程，再新增测试。
2. 每补一个模块，就尽量补齐“行为 + 边界 + 对象语义”三类测试。
3. 每修一个 bug，就补一个回归测试。
4. 优先把基础容器测试打牢，再处理高级结构。
5. 测试命名、目录结构、目标命名尽量统一。

## 十二、测试分层规划

后续测试不建议继续混成一类，而是分为三层：

### 1. ADT 契约测试

这一层只测试所有实现都应满足的公共语义，不绑定具体存储方式。

例如：

- 线性表初始化后为空
- 插入后元素数量变化正确
- 越界访问抛异常
- 删除后元素顺序正确
- 清空后恢复到初始逻辑状态

这一层适合做成模板化共享测试，复用于：

- `Sequential_List_Static`
- `Sequential_List_Dynamic`
- `Link_List_Forward`
- `Link_List_Double`

以及后续其他满足同一 ADT 的实现。

### 2. 实现特性测试

这一层只测试某种实现独有的行为，不应误当成所有实现的统一契约。

例如：

- 静态顺序表满容量后的行为
- 动态顺序表的扩容/缩容策略
- 链表实现的特定边界行为
- 堆、树、图各自的内部结构不变量

这类测试可以长期保留，但不应该写进 ADT 公共测试中。

### 3. 回归测试

每修复一个已确认 bug，就补一个最小复现测试。

这一层的作用是：

- 防止同类问题再次出现
- 把历史缺陷转化为长期质量资产

回归测试有时属于 ADT 层，有时属于具体实现层，需要按问题本身归类。

## 十二、关于“特殊行为”的处理原则

后续测试中，不应默认把当前实现的所有特殊行为都固化为长期契约。

建议区分为三类：

1. 公共接口语义
   - 这类行为应写入测试并长期稳定保证
2. 当前实现细节
   - 这类行为可以记录，但不应轻易写死到公共测试中
3. 已知不健康或待修行为
   - 不应继续作为正确行为固化到测试里

例如：

- “越界抛异常”属于公共语义
- “扩容倍率恰好是多少”更偏实现细节
- “moved-from 对象容量必须为 0” 是否合理，需要单独判断，不应默认作为统一契约

## 十三、当前阶段优先级

测试规划已经明确，但当前主优先级仍然是打通 `Data_Sturcture/C++` 的 CMake 主线。

因此，后续短期顺序为：

1. 继续扩大最小样板式的 CMake 接入范围
2. 保持现有测试可被统一构建和执行
3. 暂不立即重构测试实现细节
4. 等主线构建基本稳定后，再回头处理 ADT 契约测试与测试分层

## 十四、阶段性结论

关于测试部分，当前已经形成的明确决策是：

> 长期保留 `Boost.Test`，后续重点不是更换框架，而是把测试流程、测试覆盖和回归能力系统化。

本文件作为后续处理测试部分时的规划基线。
